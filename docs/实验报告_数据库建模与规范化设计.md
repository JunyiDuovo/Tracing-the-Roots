本报告以课程「数据库建模与规范化设计」验收要求为主线，围绕开源族谱管理系统 **寻根溯源** 的真实实现展开：阐明 **PostgreSQL** 中的实体—联系抽象与 ER 表达，给出由 ER 落地的关系模式及 **3NF／BCNF** 论证；系统整理 **主键、外键、CHECK、触发器、应用层校验** 的多层约束策略；说明 **手工录入、CSV 批量导入、Python 数据生成脚本** 等数据载入路径；节选 **递归 CTE、聚合统计** 等典型 SQL，并提示以 **客户端截图** 作为实验佐证。文末展示**个人所做工作**。

## 一、引言

### 1.1 项目背景与目标

传统纸质族谱在检索、多人协作与数据一致性校验方面成本较高。本实验关联的 Web 系统将「族谱册」电子化：支持 **用户注册／登录**、**创建多部族谱**、**邀请协作者**、**维护成员的生卒信息与血缘／配偶指针**；前端提供列表、树形预览、祖先查询、亲缘最短通路等功能，所有业务状态最终落在 **关系数据库** 中，便于持久化与复杂查询实验。

### 1.2 工程技术栈（与仓库一致）

| 层次 | 技术 | 在仓库中的典型位置 |
|------|------|---------------------|
| RDBMS | **PostgreSQL** | `sql/01_schema.sql` 及各迁移脚本 |
| 应用运行时 | Python **Flask** | `app.py` |
| ORM | **SQLAlchemy 2.x** 声明式模型 | `models.py` |
| 批量导入／生成 | **psycopg2**、`COPY`、自研脚本 | `scripts/import_member_csv.py`、`scripts/generate_bulk_data.py`、树数据生成脚本等 |
| 客户端实验 | pgAdmin／DBeaver／`psql` | 任选其一 |

**说明：** ORM 层 `models.Member` 的 `CheckConstraint` 与库内 `CHECK`、触发器互为补充；写入路径以「先通过 Flask 表单校验 → 会话提交 → DB 触发器兜底」为基本顺序。

## 二、需求到模型的映射：业务对象与建模边界

### 2.1 业务需求抽象

在不要求单独「家庭」实体的前提下，系统将以下需求结构化：

1. **身份与权限**：系统用户区别于谱内「成员」，负责登录与所有权。
2. **族谱**：逻辑上的一本谱，含谱名、主姓、修订日等书目级元数据。
3. **谱内成员**：隶属于唯一 `tree_id`（对应 `genealogy.id`）；通过 `father_id`／`mother_id` 刻画生育／谱系双亲；通过 `spouse_id` 刻画当前模型下的配偶互指。
4. **协作**：多部族谱可被多名用户协作编辑，语义为典型的 **用户—族谱 多对多**。
5. **完整性**：双亲性别、与父母出生先后顺序、生卒先后顺序等须在库内可验证（列级约束 + 行级触发器 + 可选应用层复检）。

据此得到四类 **强实体／联系表**：`app_user`、`genealogy`、`genealogy_collaborator`、`member`；其中 `member` 对 `member` 存在 **三路自引用**（父／母／配偶）。

### 2.2 「家庭」不显式建模的理由（设计取舍）

家谱领域常见两类做法：其一是建立 `Household(M:N)`／`Couple + Child`；其二是 **单亲表加外键**（本仓库采用）。后者的 **优点** 是减少了连接表与联接查询数量，表单与 CSV 结构与民间「一单一人一行」的习惯一致；**代价** 是互为配偶的成员需维护 **`spouse_id` 一致性**（系统通过应用逻辑与运维规范约束；若再婚场景复杂，可在将来引入婚配历史表——作为扩展写入报告「未来工作」即可）。

## 三、概念设计：实体、属性与联系（含基数）

### 3.1 实体与主要属性表

以下为与 `sql/01_schema.sql` **字段级对齐**的概念层描述（括号内为主要约束语义）。

#### （1）用户实体 `App_User`

| 属性 | 类型思想 | 说明 |
|------|-----------|------|
| `id` | 代理主键，自增 | 内部标识 |
| `username` | 唯一、非空 | 登录账号 |
| `password_hash` | 非空 | 口令经哈希存放，**明文密码不入库** |
| `email` | 可选 | 联系／找回（视部署而定） |
| `created_at` | 时区时间戳，默认当前 | 开户时间 |

#### （2）族谱实体 `Genealogy`

| 属性 | 说明 |
|------|------|
| `id` | 主键 |
| `title`、`surname` | 谱名、主姓氏 |
| `revision_date` | 谱册修订日（可空） |
| `created_by` | **外键** → `app_user.id`，记载创建者 |
| `created_at` | 建档时间 |

#### （3）协作者联系 `GenealogyCollaborator`

将 **用户 ⨯ 族谱** 的 **M:N** 联系提升为实体表：

| 属性 | 说明 |
|------|------|
| `(genealogy_id, user_id)` | **复合主键**兼外键两端 |
| `invited_by` | 可选，指向邀请人用户 |
| `joined_at` | 加入协作时间 |

#### （4）成员实体 `Member`

| 属性 | 说明 |
|------|------|
| `member_id` | `BIGSERIAL` 主键，支持大批量成员 |
| `tree_id` | 所属族谱，**必填**，`ON DELETE CASCADE` |
| `name`、`gender` | 姓名、性别（取值受 `CHECK` 限制） |
| `birth_date` / `death_date` | 公历精确日期（可空）；与表单、CSV 一致 |
| `birth_year` / `death_year` | **冗余**：可由触发器根据日期回填，亦可支撑仅有公元年的旧谱数据 |
| `bio` | 生平简介，`TEXT`；应用层限制 **最多 500 字符**（`app.py` 中 `_BIO_MAX_LEN`，属应用约束） |
| `generation_level` | 辈份／代数标注（可空，非负受 `CHECK` 约束） |
| `father_id`、`mother_id`、`spouse_id` | 指向 `member(member_id)`，可空，`ON DELETE SET NULL`/`ON UPDATE CASCADE`（与其它迁移脚本一致） |
| `created_by`、`created_at` | 录入者及时间 |

**联系重温（基数归纳）：**

- `App_User` **1 — N** `Genealogy`（通过 `Genealogy.created_by`）。
- `Genealogy` **1 — N** `Member`（`Member.tree_id`）。
- `App_User` **M — N** `Genealogy`（经 `genealogy_collaborator`）。
- `Member` **N — 1** `Member`（父）、**N — 1**（母）；任一子女行 **至多** 一个父指针与一个母指针。
- `Member` 与配偶：实现为可选 **互指**，课程表述可写为 **对称 1 — 1**（实现上两行互相 `spouse_id`）。

## 四、逻辑设计：ER 到关系模式的转换与范式分析

### 4.1 关系模式清单（与实现一一对应）

以下使用课程惯用 **R(属性表, 主键, 外键)** 风格概述。

1. **AppUser**(_id_, username, password_hash, email, created_at)  
   - 函数依赖：`id →` 其余所有属性；`username` 为候选键（唯一约束）。

2. **Genealogy**(_id_, title, surname, revision_date, created_by, created_at)  
   - `created_by` → `AppUser.id`。

3. **GenealogyCollaborator**(_genealogy_id_, _user_id_, invited_by, joined_at)  
   - 复合主键；`invited_by` → `AppUser.id`（可空）。

4. **Member**(_member_id_, tree_id, name, gender, birth_date, death_date, birth_year, death_year, bio, generation_level, father_id, mother_id, spouse_id, created_by, created_at)  
   - 外键：`tree_id → Genealogy.id`；`father_id, mother_id, spouse_id → Member.member_id`；`created_by → AppUser.id`。

### 4.2 冗余字段与一致性强约束

- **`birth_year`／`death_year` 与日期列并存**：属 **可控冗余**，用于（a）仅有年份的旧数据；（b）部分统计 SQL 以年份运算为主（如 `sql/03_core_queries.sql` 中寿命均值）。**一致性**由 `BEFORE INSERT OR UPDATE` 触发器在写入 `birth_date`／`death_date` 时尝试同步年份列；若仅填年份列，则日期可能为空，须理解业务允许范围。

### 4.3 范式级别论证（建议课堂陈述口径）

**定义回顾（简化）：** 1NF 要求属性原子、无重复组；2NF 消除非主属性对复合主键的部分依赖；3NF 消除非主属性对非键的传递依赖；BCNF 要求每个决定因素都是超键。

- **`app_user`**：单主键 `id`，非主属性均完全函数依赖于 `id`；`username` 上独立唯一约束，无「非键 → 非键」链依赖 **→ 满足 BCNF**。

- **`genealogy`**：单主键，所有描述谱册的属性直接依赖 `id` **→ BCNF**。

- **`genealogy_collaborator`**：主键为 `(genealogy_id, user_id)`，其余属性仅依赖整组主键（邀请人、加入时间描述该协作关系本身）**→ BCNF**。

- **`member`**：单主键 `member_id`。父／母／配偶／族谱／创建者均为 **外键引用**，在经典关系理论中记为 `member_id →` 各列，不存在「非主属性 A → 非主属性 B（且 A 非键）」的典型 3NF 反例；**在仅考虑列间函数依赖时属 BCNF**。  
  **补充说明（学术严谨性）：** 「父母出生年早于子女」涉及 **跨元组** 比较，**无法**仅由单表 `CHECK` 完全表达，故以 **触发器** 实现；这不改变上式函数依赖分析，但体现 **数据库完整性约束的分层**（列级 / 行级 / 跨行）。

## 五、约束设计：主键、外键、CHECK 与触发器

### 5.1 主键与外键策略摘要

| 表 | 主键 | 删除／更新语义（典型） |
|----|------|-------------------------|
| `member` | `member_id` | `tree_id` 引用 `genealogy(id)` **ON DELETE CASCADE**：删谱则成员级联删除，避免孤儿行 |
| `member` 自引用 | — | 父／母／配偶引用 **ON DELETE SET NULL**（见迁移 `14_member_fk_on_update_cascade.sql` 等）：删一人不强制删子女，只断开指针 |
| `genealogy_collaborator` | 复合主键 | 关联 `ON DELETE CASCADE`，随族谱或用户策略清理 |

**说明：** 工程上 `genealogy.id` 亦可能通过 `16_genealogy_fk_on_update_cascade.sql` 支持 **`ON UPDATE CASCADE`**，便于库级重编号；实验报告若执行过该脚本，须在「环境与迁移」中注明。

### 5.2 CHECK 约束（库内明示）

摘录自 `sql/01_schema.sql` 与迁移（具体边界以读者库中 `pg_get_constraintdef` 为准）：

| 约束名（示例） | 语义 |
|----------------|------|
| `ck_member_gender_csv` | 性别取值 ∈ `M,F,Male,Female,男,女`（与 CSV／国际化字段兼容） |
| `ck_member_life` | 若有生年与卒年则 **卒年 ≥ 生年** |
| `ck_member_life_dates` | 若有完整公历卒日与生日则 **卒日 ≥ 生日** |
| 列上 `CHECK` | `birth_year`／`death_year` 落在历史合理区间；`generation_level` 非负或空（以实际库为准） |

### 5.3 触发器 `trg_member_before_row`：双亲性别与年序

**动机：** `CHECK` 无法读取父母行；故在 `BEFORE INSERT OR UPDATE ON member` 上执行 PL/pgSQL 函数（`01_schema.sql`）：

- 若 `NEW.birth_date`／`NEW.death_date` 非空，回写 `NEW.birth_year`／`NEW.death_year`。
- 若存在 `father_id`：父行须存在、父 **须为男**；若父与子女均有 `birth_year`，则 **`fy < child.birth_year`**。

> 技术细节：比较使用 `>=` 判失败，即 **同年出生也不允许**（严格早于）。实验可设计「父母与子女同年」用例观察 `RAISE EXCEPTION`。

- 母亲对称：**须为女** 且 **`my < child.birth_year`**（在双方年份均存在时）。

**与业务扩展的关系：** 仓库另含 `sql/15_allow_cross_genealogy_parents.sql` 等，体现 **父母可来自其他族谱 member 行** 时仍保留性别与年序约束；应用层 `app.py` 的 `_validate_parent_refs` 亦放宽「必须同 `tree_id`」限制，与「异姓通婚」建模一致——**报告中可强调：外键指向全局 `member_id`，不限于同谱**。

### 5.4 应用层校验（Flask，`app.py`）——「DB 之外的约束」

以下 **不** 一定在 PostgreSQL 中声明，但对线上数据质量 **等效重要**，建议在实验报告「完整性分层」小节列出：

| 规则 | 实现要点 |
|------|-----------|
| 出生必填、日期合法性、不晚于今天 | `_parse_member_life_dates_from_form` |
| 中文姓名合法性 | `_validate_member_cn_name` |
| **生平长度 ≤ 500** | `_validate_bio_len`（常量 `_BIO_MAX_LEN = 500`） |
| 表单性别规范化 | `_normalize_form_gender` → `Male`/`Female` |
| 父亲≠母亲 ID、不自环为父母 | `_validate_parent_refs` |
| 父母存在且性别语义正确 | ORM 读库断言；与触发器互为双保险 |

## 九、核心 SQL、业务含义及与 Flask 实现的对应（须附本人截图）

> **截图要求：** 每题至少 **一屏**，可见 **查询文本 + 底部结果网格**（或 `\timing`/`EXPLAIN` 全文）。将占位符 **`1`** 替换为你库中真实 **存在多级后代**的 `member_id` 或 `tree_id`。

### 查询 1：成员的配偶及所有子女（`sql/03_core_queries.sql`）

**含义：** 验证 `spouse_id` 单亲互指与父母指针在 **一代范围**内的可连接性。

```sql
WITH me AS (SELECT member_id, tree_id, spouse_id FROM member WHERE member_id = 1)
SELECT 'spouse' AS rel_type, s.member_id, s.name, s.gender, s.birth_year, s.death_year
FROM me
JOIN member s ON s.member_id = me.spouse_id
WHERE me.spouse_id IS NOT NULL
UNION ALL
SELECT 'child' AS rel_type, c.member_id, c.name, c.gender, c.birth_year, c.death_year
FROM me
JOIN member c ON c.father_id = me.member_id OR c.mother_id = me.member_id;
```

**注意：** `birth_year` 列若为空但有 `birth_date`，触发器通常在写入日期时已回填年份；若以 **仅日期 CSV** 导入且跳过了触发器，需理解结果中 `birth_year` 可能为空。

### 查询 2：向上递归——所有祖先（教材式 CTE）

**含义：** 从某人沿 `father_id`／`mother_id` 向上展开**有向无环图**时，用 `path_ids` 数组防止环路；`DISTINCT ON` 取每个祖先的**最小 hop**。

```sql
WITH RECURSIVE anc AS (
    SELECT member_id, name, gender, birth_year, father_id, mother_id, 0 AS hop,
           ARRAY[member_id::bigint] AS path_ids
    FROM member WHERE member_id = 1
    UNION ALL
    SELECT p.member_id, p.name, p.gender, p.birth_year, p.father_id, p.mother_id, a.hop + 1,
           a.path_ids || p.member_id
    FROM anc a
    JOIN member p ON p.member_id = a.father_id OR p.member_id = a.mother_id
    WHERE NOT (p.member_id = ANY (a.path_ids))
)
SELECT DISTINCT ON (member_id) member_id, name, gender, birth_year, death_year, hop
FROM anc WHERE hop > 0
ORDER BY member_id, hop;
```

### 查询 3：谱内平均寿命最长的一代人

```sql
WITH life AS (
    SELECT generation_level,
           AVG((death_year - birth_year)::numeric) AS avg_lifespan_years
    FROM member
    WHERE tree_id = 1
      AND birth_year IS NOT NULL AND death_year IS NOT NULL
    GROUP BY generation_level
)
SELECT generation_level, avg_lifespan_years
FROM life
ORDER BY avg_lifespan_years DESC NULLS LAST
LIMIT 1;
```

### 查询 4：向下至多四代（可与索引实验同源）

```sql
WITH RECURSIVE down AS (
    SELECT member_id, name, father_id, mother_id, 0 AS gen
    FROM member WHERE member_id = 1
    UNION ALL
    SELECT c.member_id, c.name, c.father_id, c.mother_id, d.gen + 1
    FROM down d
    JOIN member c ON c.father_id = d.member_id OR c.mother_id = d.member_id
    WHERE d.gen < 4
)
SELECT member_id, name, gen FROM down WHERE gen > 0;
```

若在实验脚本中 **`JOIN` 附带 `AND c.tree_id = d.tree_id`**，可避免「跨界谱」在向下扩展时混入（是否需要取决于业务：`21_benchmark_indexes_four_gen.sql` 采用同谱裁剪）。

### 查询 5（补充）：网站上「祖先查询」页使用的递归 SQL（与 ORM 混用范例）

以下为 `app.py` 中 `ancestors_view` 通过 `sqlalchemy.text` 下发的语句（节选结构与参数绑定），锚点成员 **必须属于当前 `:gid`**，向上遍历时 **双亲行可落在任意 `member` 行**，与「跨谱父母」建模一致：

```sql
WITH RECURSIVE anc AS (
    SELECT member_id, name, gender, birth_date, birth_year, father_id, mother_id, 0 AS hop,
           ARRAY[member_id::bigint] AS path_ids
    FROM member WHERE member_id = :mid AND tree_id = :gid
    UNION ALL
    SELECT p.member_id, p.name, p.gender, p.birth_date, p.birth_year, p.father_id, p.mother_id,
           a.hop + 1, a.path_ids || p.member_id
    FROM anc a
    JOIN member p ON p.member_id = a.father_id OR p.member_id = a.mother_id
    WHERE NOT (p.member_id = ANY (a.path_ids))
)
SELECT DISTINCT ON (member_id) member_id, name, gender, birth_date, birth_year, hop
FROM anc WHERE hop > 0 ORDER BY member_id, hop;
```

**（1）概念与逻辑建模方面。** 我将业务对象落实为四座关系：`app_user` 管登录与所有者，`genealogy` 表征一部族谱，`genealogy_collaborator` 拆解 **用户–族谱** 的多对多协作，`member` 以 **自引用外键** 表达父母与配偶。经讨论不设独立「家庭」实体，而是用 `father_id`／`mother_id`／`spouse_id` 降低表联接复杂度，并接受互指需在应用侧维护的一致性成本。范式上我对照检查了各表：**单主键表**在非主属性不依赖其它非主属性的条件下可认定为 **不低于 3NF**；`*_collaborator` 二元联系表可归为 **BCNF**；`**member`** 上与「父母早于子女」「父母性别role」有关的规则无法仅靠 `CHECK` 表达，从而在报告中明确 **`trg_member_before_row`** 承担 **完整性约束中的跨元组层面**，与应用层 **`app.py`** 中的 `_validate_parent_refs`、生辰解析等校验形成分层。

**（2）PostgreSQL 与实验脚本。** 我在 **pgAdmin**（或等价客户端）中建库、核对 **`sql/01_schema.sql`** 与后续迁移脚本，理解 **`genealogy` 删除时成员级联**、双亲引用 **`SET NULL`/级联更新** 等语义；阅读 **`sql/02_indexes.sql`** 中 **`pg_trgm`** 支持的姓名模糊路径与 **`father_id`/`mother_id`** 上的 B-tree；并编写／使用 **`sql/21_benchmark_indexes_four_gen.sql`** 在 **四代向下递归查询**上做 **`EXPLAIN (ANALYZE, BUFFERS)`**，通过事务内 **`DROP INDEX` + `ROLLBACK`** 对照有无索引的执行计划与耗时。**`sql/03_core_queries.sql`** 中配偶／子女、`WITH RECURSIVE` 上下游与 **`app.py`** 祖先页内嵌递归 SQL（含 `birth_date` 与同谱锚点条件）我做了逐条对齐与本地化替换执行，相关内容与截图见附件。
